# 动态数据脱敏系统分工与执行计划

本文件记录本项目后续基本不变的真实分工方式，以及 Codex 在两天开发周期内的阶段方案和 TODO。除非项目负责人明确要求调整，否则后续开发、报告整理和答辩准备均以本文件为准。

## 1. 真实分工

本项目不采用传统的“前端一人、后端一人、数据库一人”分工。考虑到小组只有两名成员，并且开发时间紧张，最终采用以下分工：

```text
Codex：开发全流程主力
成员 A：项目负责人 / 监工 / 决策确认 / 最终验收
成员 B：报告整理 / 数据库设计文档 / PPT / 答辩材料
```

## 2. Codex 职责

Codex 负责完成系统开发主线，优先保证项目能够运行和演示。

主要职责：

- 拆分和整理数据库 SQL 脚本。
- 搭建 Spring Boot 后端工程。
- 搭建 Vue 前端工程。
- 实现登录认证和角色权限加载。
- 实现学生信息查询。
- 实现动态数据脱敏核心逻辑。
- 实现访问日志写入和查询。
- 实现脱敏规则查看、修改和测试。
- 实现前端核心页面。
- 联调前后端和数据库。
- 按项目宪法维护进度文档、接口契约和数据字典。

Codex 开发原则：

- 先完成可演示闭环，再补充非核心功能。
- 优先保证系统能启动、能登录、能查询、能脱敏、能审计。
- 不优先实现完整 CRUD、复杂报表、复杂异常检测和 SQL 自由查询。
- 
- 每个阶段结束后更新 `docs/progress/PROGRESS.md`。
- 每次增删改接口后更新 `docs/api/API_CONTRACT.md`。
- 每次改动数据库结构或 SQL 契约后更新 `docs/data/DATA_DICTIONARY.md`。

## 3. 成员 A 职责

成员 A 作为项目负责人和监工，不承担具体代码开发主线，主要负责决策、验收和问题反馈。

主要职责：

- 确认功能范围，避免需求继续膨胀。
- 运行或查看 Codex 交付的系统。
- 验收核心演示流程是否走通。
- 反馈页面、接口、数据或演示中的明显问题。
- 决定是否砍掉非核心功能。
- 确认最终演示账号、演示数据和演示顺序。
- 将系统截图、运行结果和关键说明交给成员 B 整理报告和 PPT。

重点验收标准：

- 后端能启动。
- 前端能启动。
- 数据库能初始化。
- 用户能登录。
- 不同角色查询同一学生数据时，脱敏结果明显不同。
- 修改脱敏规则后，查询结果能体现变化。
- 审计日志能看到查询记录。

## 4. 成员 B 职责

成员 B 负责报告、数据库设计说明和答辩材料，不参与代码主线，避免两天内沟通和集成成本过高。

主要职责：

- 编写课程设计报告。
- 整理需求分析。
- 设计数据库概念模型。
- 设计数据库逻辑模型。
- 设计数据库物理模型。
- 绘制 E-R 图。
- 整理关系模式、主键、外键和完整性约束。
- 整理索引设计、视图设计、触发器设计、存储过程设计。
- 整理数据库安全设计，包括 RBAC、动态脱敏、访问审计和异常访问检测。
- 编写测试用例和测试结果。
- 制作答辩 PPT。
- 编写答辩讲稿和演示流程。

成员 B 主要参考资料：

- `docs/product/PRD.md`
- `docs/data/DATA_DICTIONARY.md`
- `docs/api/API_CONTRACT.md`
- `数据库建表数据结构v3.md`
- `项目业务逻辑设计v2.md`
- `用户角色权限管理表v2.md`

## 5. 两天交付目标

两天内的目标不是完成一个生产级系统，而是完成一个可运行、可展示、逻辑完整的课程演示系统。

必须完成的核心闭环：

```text
初始化数据库
  -> 用户登录
  -> 加载角色和权限
  -> 查询学生信息
  -> 按角色返回不同脱敏结果
  -> 修改脱敏规则
  -> 再次查询体现规则变化
  -> 查看访问日志
```

必须支持的演示账号：

```text
admin / 123456
teacher01 / 123456
analyst01 / 123456
security_admin / 123456
auditor01 / 123456
normal_user / 123456
```

- 登录接口和登录页面。

- 脱敏规则查看和简化修改。

- 规则变更日志。
- 敏感字段列表。

### 6.3 两天内不优先做

- 完整学生成绩管理。

- 审计报告导出。
- 自动扫描敏感字段。
- SQL 自由查询框。
- 复杂异常访问检测。

这些功能可以在报告中作为完整设计或后续扩展说明。

## 8. 最终演示流程

最终答辩建议按以下顺序演示：

```text
1. admin 登录，查看原始学生数据。
2. teacher01 登录，查看轻度脱敏数据。
3. analyst01 登录，查看泛化数据。
4. normal_user 登录，查看高度脱敏数据。
5. security_admin 登录，修改手机号或身份证脱敏规则。
6. teacher01 再次查询，展示规则变化。
7. auditor01 登录，查看访问日志。
```
