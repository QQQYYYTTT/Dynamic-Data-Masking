下面是基于我们前面讨论后，重新梳理出的**最终版用户角色权限管理设计**。这个版本适用于你的项目：

> **高校学生信息动态脱敏系统**

整体采用 **RBAC 权限模型**，并且在管理员层面做进一步分工。

------

# 一、最终角色设定

系统角色分为两大类：

```text
一类是管理类角色：负责系统、数据、安全、审计等管理工作
一类是业务类角色：负责查询、教学、分析等业务使用
```

最终建议设置 8 类角色：

| 角色编码         | 角色名称      | 角色定位                                 |
| ---------------- | ------------- | ---------------------------------------- |
| `SUPER_ADMIN`    | 超级管理员    | 系统最高权限，拥有全部权限               |
| `SYSTEM_ADMIN`   | 系统管理员    | 管理用户、角色、权限等系统基础配置       |
| `DATA_ADMIN`     | 数据管理员    | 管理学生数据资源、字段、数据质量         |
| `SECURITY_ADMIN` | 安全管理员    | 管理敏感字段、脱敏规则、安全策略         |
| `TEACHER`        | 教师/教务老师 | 查询学生信息，用于教学和教务管理         |
| `ANALYST`        | 数据分析师    | 使用脱敏或泛化数据进行统计分析           |
| `AUDITOR`        | 安全审计员    | 查看访问日志、规则变更日志、异常访问记录 |
| `NORMAL`         | 普通用户      | 低权限用户，只能查看有限的高度脱敏数据   |

------

# 二、区分

这四个角色最容易混乱：

| 角色                        | 一句话定位 | 主要职责                                                 | 不应该拥有的权限                   |
| --------------------------- | ---------- | -------------------------------------------------------- | ---------------------------------- |
| `DATA_ADMIN` 数据管理员     | 管数据     | 维护学生数据、管理字段、导入导出基础数据、保证数据完整性 | 不配置脱敏规则、不审计他人访问     |
| `ANALYST` 数据分析师        | 用数据     | 使用脱敏/泛化数据做统计分析、生成报表                    | 不修改学生数据、不查看原始敏感数据 |
| `SECURITY_ADMIN` 安全管理员 | 配规则     | 配置敏感字段、脱敏策略、安全规则                         | 不直接查询学生原始业务数据         |
| `AUDITOR` 安全审计员        | 查行为     | 查看访问日志、脱敏命中日志、规则变更日志、异常访问       | 不修改规则、不查询业务数据         |

可以总结成：

```text
数据管理员：把数据管好
数据分析师：把数据用好
安全管理员：把规则配好
安全审计员：把行为查好
```

# 三、五张用户角色权限管理表

系统采用五张核心权限表：

```text
1. sys_user              用户表
2. sys_role              角色表
3. sys_permission        权限表
4. sys_user_role         用户角色关系表
5. sys_role_permission   角色权限关系表
```

关系如下：

```text
sys_user
   ↓
sys_user_role
   ↓
sys_role
   ↓
sys_role_permission
   ↓
sys_permission
```

也就是：

```text
用户 User  ←→  角色 Role  ←→  权限 Permission
```

# 四、用户表 `sys_user`

## 1. 表作用

保存系统登录用户信息。

用户可以是：

```text
管理员
教师
数据管理员
安全管理员
数据分析师
安全审计员
普通用户
```

## 2. 表结构

```sql
CREATE TABLE sys_user (
```

# 六、权限表 `sys_permission`

## 1. 表作用

权限表定义系统中所有可控制的操作。

建议分为三类：

| 权限类型 | 含义         | 示例                                   |
| -------- | ------------ | -------------------------------------- |
| `MENU`   | 前端页面权限 | 是否能看到脱敏规则管理页面             |
| `API`    | 后端接口权限 | 是否能调用新增用户接口                 |
| `DATA`   | 数据访问权限 | 是否能查看原始数据、是否能查询敏感字段 |

## 2. 表结构

```sql
CREATE TABLE sys_permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '权限ID',

    permission_code VARCHAR(100) NOT NULL UNIQUE COMMENT '权限编码',
    permission_name VARCHAR(100) NOT NULL COMMENT '权限名称',

    permission_type VARCHAR(20) NOT NULL COMMENT '权限类型：MENU菜单，API接口，DATA数据权限',

    parent_id BIGINT DEFAULT 0 COMMENT '父权限ID',
    path VARCHAR(255) COMMENT '前端路由或后端接口路径',
    method VARCHAR(20) COMMENT '请求方法：GET、POST、PUT、DELETE等',

    description VARCHAR(255) COMMENT '权限描述',

    status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1启用，0禁用',

    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT='权限表';
```

# 八、用户角色关系表 `sys_user_role`

## 1. 表作用

绑定用户和角色。

## 2. 表结构

```sql
CREATE TABLE sys_user_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',

    user_id BIGINT NOT NULL COMMENT '用户ID',
    role_id BIGINT NOT NULL COMMENT '角色ID',

    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',

    UNIQUE KEY uk_user_role (user_id, role_id),

    CONSTRAINT fk_user_role_user
        FOREIGN KEY (user_id) REFERENCES sys_user(id)
        ON DELETE CASCADE,

    CONSTRAINT fk_user_role_role
        FOREIGN KEY (role_id) REFERENCES sys_role(id)
        ON DELETE CASCADE
) COMMENT='用户角色关系表';
```

## 3. 示例数据


# 十二、和动态脱敏系统的衔接

权限管理模块主要决定：

```text
用户是谁
用户是什么角色
用户能访问哪些页面
用户能调用哪些接口
用户能不能查学生数据
用户能不能查敏感字段
用户能不能看原始数据
```

脱敏规则模块决定：

```text
某个角色查询某个字段时，应该怎么展示
```

例如：

| 角色          | phone 字段    | id_card 字段         | family_income 字段 |
| ------------- | ------------- | -------------------- | ------------------ |
| `SUPER_ADMIN` | 原始值        | 原始值               | 原始值             |
| `TEACHER`     | `138****5678` | `510104********2345` | `80000-90000`      |
| `ANALYST`     | `138****5678` | `******************` | `80000-90000`      |
| `NORMAL`      | `***********` | `******************` | `*****`            |
| `DATA_ADMIN`  | 按规则脱敏    | 按规则脱敏           | 按规则脱敏         |

系统执行逻辑可以写成：

```text
1. 用户登录
2. 查询用户角色
3. 查询角色权限
4. 判断是否有接口访问权限
5. 判断是否有数据访问权限
6. 查询数据库原始结果
7. 如果有 DATA:VIEW_RAW，返回原始数据
8. 如果只有 DATA:VIEW_MASKED，根据角色加载脱敏规则
9. 返回脱敏结果
10. 写入访问审计日志
```

# 十三、总结

> 本系统采用基于角色的访问控制模型 RBAC 设计用户角色权限管理模块。系统中用户不直接绑定权限，而是通过角色间接获得权限。权限分为菜单权限、接口权限和数据权限三类，分别用于控制前端页面访问、后端接口调用和数据访问粒度。
>
> 针对高校学生信息动态脱敏场景，系统设置超级管理员、系统管理员、数据管理员、安全管理员、教师、数据分析师、安全审计员和普通用户八类角色。其中，超级管理员拥有全部权限；系统管理员负责用户、角色和权限管理；数据管理员负责学生数据资源维护；安全管理员负责敏感字段和脱敏规则配置；教师负责教学和教务查询；数据分析师使用脱敏或泛化数据进行统计分析；安全审计员负责查看访问日志和规则变更记录；普通用户只能查看高度脱敏后的有限数据。
>
> 通过该设计，系统实现了数据管理、数据使用、安全配置和安全审计之间的职责分离，避免同一用户既能修改规则又能审计规则，或既能配置规则又能随意查看原始敏感数据，从而体现最小权限原则和动态脱敏场景下的安全访问控制需求。
