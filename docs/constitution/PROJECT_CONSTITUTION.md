# 项目宪法

本文件定义“动态数据脱敏系统”的开发规则、目录约定和文档维护责任。后续人类开发者和 Codex 都应遵守本文件。

## 1. 强制文档维护规则

以下规则是强制约定：

- Codex 每次开发会话结束前，必须更新 `docs/progress/PROGRESS.md`。
- 每当新增、删除、重命名接口，或修改接口请求参数、响应结构、权限要求、审计行为时，必须在同一次会话中更新 `docs/api/API_CONTRACT.md`。
- 每当修改数据库表、字段、关系、索引、视图、存储过程、触发器或种子数据契约时，必须在同一次会话中更新 `docs/data/DATA_DICTIONARY.md`。
- 后续协作文档统一使用中文。

## 2. 需求基线

项目当前需求基线来自三份根目录文档：

- `项目业务逻辑设计v2.md`
- `用户角色权限管理表v2.md`
- `数据库建表数据结构v3.md`

开发执行时优先参考：

- `docs/product/PRD.md`
- `docs/api/API_CONTRACT.md`
- `docs/data/DATA_DICTIONARY.md`

如果原始文档和执行文档冲突，应先更新执行文档并在进度文档中记录决策。

## 3. 架构方向

- 后端采用 Java 技术栈，计划基于 Spring Boot 构建。
- 前端采用 Vue 3 技术栈，面向管理后台和课程演示场景。
- 数据库以 MySQL 8 为主。
- 权限认证计划使用 JWT + Spring Security。
- 数据库访问计划使用 MyBatis。
- 密码必须使用 BCrypt 等安全哈希方式存储，不得保存明文密码。
- 前端不得接收明文敏感数据后再自行脱敏。脱敏应在后端或数据库侧完成后再返回给前端。

## 4. 后端目录约定

后端代码放在 `backend/src/main/java/com/ddm` 下。

- `common/config`：框架配置和应用配置。
- `common/exception`：全局异常和错误处理。
- `common/response`：统一响应对象。
- `common/security`：认证、授权和安全相关工具。
- `common/util`：通用工具类。
- `module/auth`：登录、登出、令牌、当前用户等认证相关功能。
- `module/user`：用户、角色、权限和 RBAC 管理。
- `module/student`：学生信息和学生成绩查询与管理。
- `module/masking`：敏感字段、脱敏方式、脱敏策略、规则分配和脱敏算法。
- `module/audit`：访问日志、规则变更日志和异常访问记录。
- 后续如实现报表，可新增 `module/report`。

业务模块应保持职责分离：

- `controller`：接口入口，只做参数接收、权限入口声明和响应包装。
- `service`：业务逻辑、事务控制和跨表协作。
- `mapper`：数据库访问。
- `entity`：数据库实体。
- `dto`：请求与响应对象。
- `strategy`：脱敏策略实现。

## 5. 前端目录约定

前端代码放在 `frontend/src` 下。

- `api`：按后端模块划分接口请求函数。
- `router`：路由定义和路由守卫。
- `stores`：全局状态管理。
- `layout`：系统主布局。
- `views`：页面级视图。
- `components`：可复用组件。
- `utils`：通用工具函数。
- `styles`：全局样式和主题变量。

前端页面以实际业务操作为主，不做无意义的营销型落地页。

## 6. 权限约定

系统使用 RBAC 权限模型。

权限分为三类：

- `MENU`：控制前端页面是否可见。
- `API`：控制后端接口是否可调用。
- `DATA`：控制数据访问范围和数据明文/脱敏粒度。

角色边界：

- `SYSTEM_ADMIN` 只管理用户、角色、权限，不查看学生业务数据。
- `DATA_ADMIN` 管理学生数据，不配置脱敏策略。
- `SECURITY_ADMIN` 配置敏感字段和脱敏规则，不查询学生业务数据。
- `ANALYST` 使用脱敏或泛化数据，不查看原始敏感数据。
- `AUDITOR` 查看审计和异常记录，不修改规则和业务数据。

## 7. 接口约定

- 接口路径统一以 `/api` 开头。
- 接口按模块划分，例如 `/api/auth`、`/api/student`、`/api/masking`、`/api/audit`、`/api/system`。
- 后端应使用统一响应结构。
- 所有受保护接口必须在 `docs/api/API_CONTRACT.md` 中声明所需角色或权限。
- 涉及敏感数据查询、导出、规则变更、异常处理的接口必须声明审计行为。

## 8. 数据约定

- 表名使用小写下划线命名。
- 字段名使用小写下划线命名。
- 主键必须明确且稳定。
- 关系表命名应清晰表达两端实体，例如 `sys_user_role`。
- 敏感字段必须在数据字典中记录敏感等级和默认脱敏方式。
- 用于课程展示的数据库特性，例如视图、函数、存储过程、触发器和索引，必须在数据字典中记录。
- 业务查询返回给前端的字段如与数据库字段命名不同，应在接口契约中体现。

## 9. 脱敏约定

- 脱敏策略命名应保持一致，例如 `NO_MASK`、`FULL_MASK`、`KEEP_PREFIX`、`KEEP_SUFFIX`、`KEEP_PREFIX_SUFFIX`、`EMAIL_MASK`、`ADDRESS_LEVEL`、`GENERALIZATION`、`HASH_MASK`、`KEEP_YEAR`、`NOISE`。
- 最终脱敏效果应由用户数据权限、用户角色、敏感字段和策略分配共同决定。
- 脱敏策略优先级为：原始数据权限 > 角色分配策略 > 字段默认策略 > 安全兜底。
- 审计可复现的场景应优先使用确定性脱敏；只有策略明确允许时才使用随机脱敏或加噪。

## 10. 审计约定

以下行为必须记录：

- 登录。
- 查询学生信息或学生成绩。
- 导出学生数据、统计报表或审计报告。
- 新增、修改、删除、启用、禁用脱敏策略。
- 触发异常访问检测或处理异常访问记录。

访问日志应尽量记录：

- 用户 ID、用户名、角色编码。
- 操作类型。
- 查询条件或 SQL 摘要。
- 访问表和字段。
- 涉及的敏感字段。
- 是否执行脱敏。
- 脱敏策略快照。
- 返回记录数。
- 客户端 IP。
- 访问时间。

## 11. 进度记录约定

每次开发会话结束时，`docs/progress/PROGRESS.md` 应记录：

- 日期和会话摘要。
- 已完成事项。
- 修改过的文件或区域。
- 本次做出的决策。
- 已知问题。
- 建议下一步。

## 12. 变更纪律

- 每次改动应聚焦当前任务，避免无关重构。
- 不删除用户明确要求保留的三份根目录设计文档。
- 不在未确认前引入新的大型框架或复杂架构。
- 优先保证结构清晰、逻辑严密、适合课程展示。
